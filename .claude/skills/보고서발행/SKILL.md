# 보고서 발행 세션 SKILL

> **이 스킬은 「발행 세션」 전용이다.** 분석 세션이 만들어 둔 보고서 초안·로컬 커밋을 받아 **허브 등재·전편 링크·문서 갱신·push**를 일괄 처리한다.
> **새 분석·새 수집·수치 재계산은 절대 이 세션에서 하지 않는다** — 그건 `보고서분석` 스킬이 담당한다. (세션 분리 원칙: 분석과 발행은 한 세션에 섞지 않는다.)

## 이 스킬이 담는 것 / 담지 않는 것

- **담는 것(기계적 반복):** 보고서 md 골격, 발행 배치 루틴, git 규율, 발행 후 렌더링 검증.
- **담지 않는 것(판단):** 해석 문장·한 줄 결론의 최종 확정, 발행 승인. → **⛔ 정지선**에서 회장님이 결정한다.

---

## 1. 시작 점검

1. **작업 폴더 확인**: `문서\역량부서\인천항물동량` 인지 확인(⚠️ `수출서류랩`과 섞지 않는다).
2. **인수인계 §7 확인**: `docs/다음세션_인수인계.md` §7 "다음 작업"이 이번 발행과 일치하는지 확인.
3. **대기 커밋 확인**: `git log --oneline -4`, `git status`로 분석 세션이 남긴 로컬 커밋(초안)과 origin 대비 ahead 수를 확인한다.

## 2. 보고서 md 골격 (#01~#03 공통 구조)

새 보고서 `reports/report_NN_<주제>.md`는 아래 순서를 그대로 따른다.

1. **제목 (결론형)**: `# 보고서 #NN — <핵심 수치>: <의미 부제>` (예: "인천항 빈 컨테이너의 85.1%는 수출 방향: 불균형의 방향 규명")
2. **전편 연결 인용블록**: `> [보고서 #(NN-1)](...)`로 이어 붙이는 한 문단. 앞 보고서가 무엇을 했고 이번이 무엇을 닫는지.
3. **메타 3줄**: `- **작성일**:` / `- **분석 대상**:` / `- **한 줄 결론**:`
4. **`## 1. 핵심 요약`** — 불릿 2~3개(핵심 수치 + 체인 완결 문장).
5. **`## 2. 분석 결과`** — **차트(`![...](images/...png)`) → 출처/한계 `<sub>` 캡션 → 표(제목 `### 표 N.` + 표 + 반올림 오차 `<sub>` 각주)** 순서.
6. **`## 3. 해석`** — 맨 위 **디스클레이머 인용블록**(`> 아래 해석은 이번 데이터 범위 안에서의 추론이며, 단정이 아니라 설명 가설로 제시한다.`) + `### 3-1 / 3-2 / 3-3` 소제목.
7. **`## 4. 방법론`** — 이번 보고서의 핵심 난제와 해결.
8. **`## 5. 한계 및 후속 과제`** — 한계 + 후속 과제 불릿.
9. **`## 부록: 데이터 및 재현`** — `### 사용 데이터`(표) / `### 사용 기술` / `### 재현 방법`(코드블록) / `### 개발 참고`.
   - 수정이 발생하면 부록 끝에 `### 수정 이력` 을 두고 `- YYYY-MM-DD: <내용> (수치 무관)` 형식으로 1줄씩 남긴다.

## 3. 작성일 규칙

- **실제 작성 연도**를 쓴다. 오늘 날짜 기준.
- ⚠️ **과거 오기 사고**: `2025-07-08`처럼 연도를 한 해 잘못 적어 `2026`으로 정정한 실례가 있다. "분석 대상의 2025년"(데이터 연도)과 "작성일 연도"를 혼동하지 말 것. 작성일은 반드시 오늘 연도로 재확인한다.

## 4. 🔒 삭제 불가 항목 — 개발 참고(AI 활용 명시)

부록 `### 개발 참고`에는 **어떤 경우에도** 다음 두 가지를 남긴다:
1. **AI 도구(Claude Code) 활용 사실 명시.**
2. **직접 수행한 범위 명시** — 최소한 **데이터 선정·검증·해석**은 사람이 했다는 문장.

실제 문장 예(#01): "개발 과정에서 코드 작성과 디버깅에 AI 도구(Claude Code)를 활용했다. 다만 **어떤 데이터를 쓸지 선정하고, 후보 API를 직접 호출해 목적에 맞는지 검증하고, 결과를 해석하는 판단은 직접 수행했다.**"
→ 이 항목은 포트폴리오 정직성의 핵심이므로 **분량을 줄이더라도 빼지 않는다.**

## 5. ⛔ 정지선 — 해석 확정과 통독

1. **해석 문장·한 줄 결론은 회장님 확인 후 확정한다.** 스킬이 임의로 확정하지 않는다.
2. **push 전, 회장님이 초안 전문을 직접 통독한다.**
   - ⚠️ **md 에디터 자동 포맷이 링크를 지운 사고 실례**가 있다(출처 각주 링크 누락 → 별도 커밋으로 보완). 그러므로 push 전 전문 통독은 **필수**다. 특히 `[텍스트](경로)` 링크·`<sub>` 캡션·표 각주가 살아 있는지 눈으로 확인한다.

## 6. 발행 배치 루틴 (#03 발행 실제 순서)

지정된 순서를 그대로 따른다. **보고서의 수치·표·차트는 절대 수정하지 않는다** — 아래는 문구·링크·목록 편집만.

1. **README 허브 등재**:
   - "보고서 목록"에 신규 항목 추가. 링크 라벨은 **"보고서 전문 보기 →"** 유지(파일 경로 노출 금지).
     ```
     - **[보고서 #NN] <제목> (<연도>)** — <한 줄 요약>.
       → [보고서 전문 보기 →](reports/report_NN_<주제>.md)
     ```
   - 목록 아래 **체인 인용블록** 갱신(#01→…→#NN을 하나의 연구 흐름으로 잇는 문장).
   - **저장소 구조 트리**의 `reports/` 아래에 새 파일 한 줄 추가(트리 문자 `├──`/`└──` 정렬 유지).
2. **전편 보고서에 후속 링크 배치**: 앞 보고서(들)의 해당 위치(예: §4 한계·후속, §5 후속 과제)에 `[보고서 #NN](...)` 후속 링크를 같은 스타일로 이어 붙인다. 필요 시 용어 통일도 이 배치에 함께(수치 무관).
3. **`docs/TODO.md` 완료 처리**: 이번 발행으로 닫힌 항목을 `⬜ → ✅` 로 바꾸고 완료일(YYYY-MM-DD)·수행 위치를 표기한다.
4. **`docs/다음세션_인수인계.md` 갱신**:
   - **§2 보고서 표**에 신규 행 추가(핵심 결론 수치).
   - **§6 작업 기록**에 오늘 날짜 발행 항목 추가.
   - **§7 다음 작업**을 다음 세션 내용으로 교체.

## 7. git 규율

1. **지정 파일만 스테이징**한다(발행 배치에서 실제 편집한 파일만; `git add .` 금지).
2. **`git diff`에서 숫자가 바뀐 줄이 하나라도 있으면 정지·보고.** 보고서 수치·표·차트는 무변경이어야 한다. 새로 추가한 안내 문구의 숫자가 기존 확정치와 일치하는지도 확인.
3. **커밋 1건**. 메시지 형식(#03 발행 커밋 참고):
   ```
   보고서 #NN 발행: README 허브 등재 + #(전편들) 후속 링크·용어 정리
   ```
4. **push**: `git push origin main`.
   - **2분 타임아웃** 시 커밋까지만 하고 멈춘 뒤 **별도 PowerShell 창에서 `git push origin main`** 을 직접 실행한다(기존 루틴).
   - push 성공 시 `git status`(ahead 0 확인)·`git log --oneline -4`로 마무리 보고.

### 정지선 요약 (스킬이 자동으로 넘지 않는 지점)

1. **§5-1** — 해석·한 줄 결론 최종 확정 전 정지(회장님 확인).

### 🔒 절대 불변

- 보고서의 **수치·표·차트는 발행 세션에서 수정하지 않는다.**
- 부록 **개발 참고(AI 활용 + 직접 수행 범위)** 는 삭제하지 않는다.
